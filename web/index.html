HOW IT WORKS

Seven steps.
One command.

01
You give it a goal
Plain English. No XPaths, no CSS selectors, no waits. "Log in, search for headphones, and add the cheapest to cart."
02
It opens a browser
Real Chromium via Playwright. Headed by default so you can watch, headless when you want speed.
03
It reads the page
Every step it extracts a structured DOM snapshot — clickables, inputs, headings, errors. No screenshots sent to the LLM.
04
The AI decides what to do next
Gemini, Claude, or OpenAI receive the page state plus the goal. They return one or more actions: TYPE, CLICK, SCROLL, OBSERVE, BATCH.
05
It acts and observes
Skopix executes the action, captures a screenshot, waits for the page to settle, then loops back to step 3 with the new state.
06
It flags issues as it sees them
Broken images, console errors, slow loads, broken auth — anything that shouldn't be there gets recorded with severity, type, and screenshot.
07
You get a full report
HTML report with every step, screenshot, video recording, and issue list. Issues automatically pushed to GitHub, Jira, or Linear — without duplicates.
THE DASHBOARD

A full UI
for your QA team.

Skopix isn't just a CLI. The dashboard runs locally on port 9000 and gives you a complete visual workspace for managing tests, suites, runs, credentials, and issues. Open it any time with skopix dashboard.

Live Dashboard
Test stats at a glance — pass rate, total runs, issues found, average duration. Recent sessions list with one-click access to reports and re-runs.
All Tests view
Universal flat table showing every test across every suite plus standalone tests. Search by name or goal, filter by scope, tags, or credentials. Sortable columns.
Test suites
Group related tests into suites. Run them all in one click — sequentially or in parallel. Suite-level defaults for URL, credentials, headless mode, and trackers.
Suite runs history
Every suite execution recorded with grouped pass/fail bar visualisation. Click in to see per-test results, durations, status, and direct links to individual session reports.
Live "Running" view
When a test or suite is active, a Running tab appears with a pulsing indicator. Watch the agent work in real time with a 520px-tall console. Copy output before navigating away.
Credentials vault
Save logins by name in ~/.skopix/credentials.yaml with restricted file permissions. Tests reference them by name only — values never appear in test files, logs, or screenshots.
Issues view
Every issue raised across all tests. Filter by status, tracker, severity. See occurrence counts. Sync from GitHub/Jira/Linear with one click. Mark resolved manually or close on tracker.
Unified test editor
One modal handles every test — standalone or in a suite. Set URL, goal, credentials, max steps, tags, trackers (GitHub/Jira/Linear), headless mode. "Save and run" runs immediately.
Batch operations
Tick checkboxes on sessions or suite runs to bulk delete with the floating action bar. Cascade option to delete sessions linked to a suite run together.
Move & reorganise
Drag tests between suites. Move from "saved" to a suite or vice versa. Rename suites without breaking references. Duplicate any suite as a starting point.
Rich session reports
Every test run produces a standalone HTML report with screenshots per step, video recording of the full session, AI reasoning, action results, and detected issues with severity.
One-click re-run
Re-run any past session with the same goal and credentials. Useful for verifying fixes or re-checking flaky behaviour.
TEAM MODE

Built for
QA teams.

Skopix runs as a single tool you self-host. Anyone you invite joins the same workspace — sharing tests, suites, reports, and issues in real time. Each user has their own login, their own API tokens, and their own role. Single-user mode is also fully supported — team mode is opt-in via --team flag.

Secure login & sessions
Email + password authentication with scrypt-hashed passwords (no plaintext, no third-party auth dependency). HTTP-only session cookies with 30-day expiry. Sessions invalidated when accounts are disabled or removed.
Invite teammates with one link
Admins generate invite links from the Users page. New members visit the link, set their own name and password — they're auto-logged in and joined to the workspace. No SMTP required.
Three roles: admin, editor, viewer
Admin manages users, full access. Editor creates & runs tests, manages suites. Viewer is read-only — perfect for QA leads and stakeholders. Write controls are hidden from viewers entirely (not just disabled).
Run attribution
Every test run records who triggered it. Reports show "Run by Adil · 14:32". Audit log captures logins, role changes, deletions, and every administrative action. Know exactly what happened and who did it.
Real-time shared workspace
Tests run on the server, not the user's machine. When Bob runs a suite, Alice sees it in her dashboard immediately. Reports, suite runs, and issues are all shared instantly across the team.
SQLite database
User accounts, sessions, invites, and audit logs stored locally in a single SQLite file at ~/.skopix/skopix.db. No Postgres setup. No external services. Move it with the rest of your data.

PER-USER TOKENS

Your API keys.
Your tokens.

Workspace-wide tokens in .skopix.env work as defaults for the whole team. Individual users can override with their own — encrypted at rest, never visible to anyone else, even admins.

AES-256-GCM encrypted
Every stored token is encrypted with a key derived from SKOPIX_SECRET_KEY. Only the decryption layer ever sees plaintext. If someone gets your database file, they get encrypted bytes — not your tokens.
User-first, workspace fallback
When a test runs, Skopix checks for the user's personal token. If found, uses it. If not, falls back to the workspace default. Same dashboard, same suite, but Bob's GitHub issues are created as Bob, and Alice's as Alice.
Private to each user
Admins can see which tokens a user has set, but never the values. No "god mode" — even the workspace owner can't read other users' personal credentials. Each user manages their own from their Settings page.
Supported integrations
GitHub token, Jira email + API token, Linear API key. Gemini, Claude, OpenAI API keys (so each user can use their own LLM quota). Workspace base URLs and project keys stay shared because those are team-level.
PARALLEL TESTING

Run up to 5
tests at once.

Slow tests are dead tests. Skopix supports running multiple tests in parallel, each in its own browser instance with full isolation. Configure your suite's "Max parallel tests" slider in the dashboard, or use --parallel N from the CLI.

1×
Sequential (default)
Run tests one at a time. Easiest to follow with a clean console. Best for small suites or when you're watching the agent live.
3×
Recommended
Three workers running simultaneously. Best balance of speed and resource use for typical setups. Console output is tagged [T1], [T2], [T3] so you can follow each.
5×
Maximum throughput
Five workers — best for large suites on a paid LLM tier with at least 16GB RAM. Watch out for rate limits on the free Gemini tier.
SMART ISSUE DEDUPLICATION

Never duplicate
a bug ticket again.

Most QA tools spam your tracker with duplicates every time a flaky bug recurs. Skopix uses a three-stage pipeline to detect when an issue has already been raised — and comments on the existing ticket instead of creating a fresh one. Closed tickets respected as resolved (regression signal).

Stage 1 — Fingerprint match
Each issue gets a deterministic SHA-256 fingerprint based on suite name, test name, URL hostname/path, type, and a normalised title. Identical phrasing matches instantly.
Stage 2 — URL-scoped filter
"Failed to load resource" on Site A and Site B should be different tickets. Skopix filters candidate issues to those with the same hostname AND pathname before semantic comparison.
Stage 3 — Semantic match (LLM)
If candidates exist, Gemini Flash Lite compares the new issue against them. "Broken images" vs "Images failed to load" vs "404 errors on image resources" — all recognised as the same bug.
Auto-sync from trackers
Before adding a comment, Skopix verifies the issue is still open on GitHub/Jira/Linear. If it's been closed, that's a regression — Skopix creates a fresh ticket.
Local issue store
Records kept in skopix-reports/.issues/issues.json with fingerprint, tracker ref, status, occurrence count, and last 20 session IDs. Source of truth for dedup.
Recurrence comments
When a known issue recurs, Skopix adds a comment with timestamp, suite, test, and session ID. Your team sees how often it's happening — useful signal for prioritisation.

FEATURES

Everything you need
to ship faster.

Multi-LLM support
Gemini (free tier supported), Claude, and OpenAI. Switch per-test or per-suite. Same agent loop, same DOM extraction — just swap the brain.
Headless or headed
Watch the agent work in a real browser, or run silently in headless mode for CI. Toggle per-test or per-suite. Headless is faster and uses less RAM.
Goal-based pass/fail
Test passes if the goal is achieved. Issues found during the test are reported separately as observations — they don't fail the test unless the goal itself failed.
Auto-detect login flows
Skopix recognises login goals and auto-detects success (post-login content, URL change) or failure (error messages, still on login page) — stops early instead of looping for 20 steps.
Smart issue flagging
The agent flags broken images, JS errors, console errors, slow loads, broken links, missing elements, accessibility issues — with severity, type, screenshot, and step number.
Tag-based organisation
Tag tests with smoke/regression/critical/whatever. Filter All Tests by tag. Build smoke-test suites by selecting tagged tests.
Video recording
Every session records video of the browser. Watch back to see exactly what happened. Useful for debugging flaky tests or understanding agent behaviour.
DOM-first, screenshots later
The LLM receives structured DOM data (clickables, inputs, headings, errors) — not screenshots. Faster, cheaper, more reliable than vision-based agents. Screenshots saved for the report only.
Resilient to changes
No CSS selectors to maintain. The agent re-evaluates the page on every step. UI redesigns won't break tests as long as the goal is still achievable.
SELF-HOST

Run it
your way.

Skopix is genuinely self-hosted. No telemetry, no phone-home, no SaaS dependency. Your data stays on your hardware. Deploy how you like — laptop, NAS, office server, cloud VM, or Docker.

Solo developer · laptop
Just run skopix dashboard. Default single-user mode, no login, no setup. Perfect for personal projects, side hustles, freelance QA work. Zero ceremony.
Small team · spare laptop or NAS
For 2-10 person QA teams. An old MacBook in the office, a Synology NAS, a Raspberry Pi 5 — anything that stays on. Run with --team --host 0.0.0.0 and your team connects at http://<ip>:9000.
Bigger team · cloud VM
Hetzner Cloud (£4/mo), DigitalOcean, Fly.io, Railway, AWS EC2 — any small Linux VM. 2GB RAM minimum for browser automation. Bind to 0.0.0.0, put behind nginx/Caddy with HTTPS, you've got a proper internal QA platform.
Docker · one command
Multi-arch image (amd64 + arm64) published to GitHub Container Registry. Pull, mount a volume, set your secret key — running in 90 seconds. Includes Caddy + auto-HTTPS recipe for production deployments.
Your data, your backups
Everything lives in two places: ~/.skopix/skopix.db (users, sessions, secrets) and ./skopix-reports/ (reports, videos, issues). Back both up, you've backed up Skopix. Move them to a new server, you're up and running.
No telemetry, no calls home
Skopix never contacts our servers. Page DOM goes to your chosen LLM provider, issues go to your tracker, that's it. We don't even know you're running it. Audit the code on GitHub if you want — it's all open.

      SKOPIX_SECRET_KEY: ${SKOPIX_SECRET_KEY}

    image: caddy:2
    restart: unless-stopped
    ports: ["80:80", "443:443"]
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data
volumes:
  caddy_data:

# Caddyfile (replace with your domain)
skopix.yourcompany.com {
    reverse_proxy skopix:9000
}

$ docker compose up -d
Point DNS at your server, run the command, visit https://your-domain. Caddy fetches a Let's Encrypt cert automatically — no certbot, no cron, no manual renewal. Full guide: DOCKER.md.

FAQ

Common questions.

How is this different from Playwright/Cypress/Selenium? +
Traditional tools require you to write deterministic scripts: "click button with ID #submit". Skopix uses an LLM to navigate based on a goal. There are no selectors to maintain — when your UI changes, Skopix adapts. It's slower and probabilistic, but ideal for flows where the exact path varies (signups, checkout, multi-step wizards) and for catching UX issues humans would notice (broken images, slow loads, confusing copy).
How much does it cost to run? +
With Gemini's free tier — £0. The free tier allows 15 requests per minute, enough for a few sequential tests. For paid tiers: each test typically uses 3-15 LLM calls. On Gemini Pro that's fractions of a penny per test. Claude Sonnet is more expensive (~5p per test). The dashboard, browser automation, reports, video recording — all free.
Where are my credentials stored? +
In ~/.skopix/credentials.yaml with file permissions set to 0600 (your user only). They never leave your machine. Tests reference them by name — the values are passed to the agent at run time via a temp file that's deleted after 60 seconds. Suite YAMLs are safe to commit to git: only credential set names appear, never values.
Will it duplicate issues every time a test fails? +
No. Skopix has a three-stage dedup pipeline: exact fingerprint match → URL-scoped filter (same site & page only) → LLM semantic comparison. Recurring issues comment on the existing tracker ticket instead of creating duplicates. If you close the ticket on GitHub/Jira/Linear, the next detection creates a fresh ticket — that's a regression signal worth knowing about.
Can I run tests on different sites without them merging? +
Yes. The dedup filter requires same hostname AND same pathname before any semantic comparison happens. "Broken images" on Site A and "Broken images" on Site B will produce two separate tickets. Even within a site, "Broken image on /checkout" and "Broken image on /product" stay separate.
How does parallel testing work? +
Set "Max parallel tests" on a suite (1-5) or pass --parallel N to skopix suite. Each worker spawns its own browser instance and own LLM session. Console output is tagged [T1], [T2], etc. Recommended: 3 for most setups. Going higher requires more RAM and a paid LLM tier (free Gemini = 15 RPM limit).
Can I run Skopix in CI/CD? +
Yes. Use the CLI in headless mode with --headless. API keys and tracker creds via env vars. The exit code is 0 on pass and 1 on fail, so it integrates with any CI runner. Suite results are written to skopix-reports/ as JSON and HTML — upload as artifacts. Issues auto-pushed to GitHub/Jira/Linear during the run.
What about negative testing? +
Skopix handles negative testing through clear goal phrasing. Instead of "test invalid login fails", write "attempt to log in with invalid credentials and verify the error message appears". The error appearing IS the success criterion. The agent reads goals literally — if the goal says "verify successful login" and login fails, the test correctly fails too.
Does Skopix work with single-page apps (React, Vue, etc.)? +
Yes. The agent uses Playwright's adaptive waits — it waits for the DOM to settle after each action before re-evaluating. Works with React, Vue, Angular, Svelte, vanilla JS, anything that runs in a browser. The DOM extractor reads computed state, not source markup, so client-side rendering is invisible.
What if the LLM does something unexpected? +
Each test has a max-step limit (default 20). The agent stops itself when it decides the goal is achieved or impossible. Auto-detect rules catch common cases (login success/failure) and break the loop early. Every step is recorded with reasoning, so even when the agent goes off-track, the report shows exactly why. Issues with the agent's behaviour can be reported back via the dashboard's thumbs-down button.
Is my data sent to third parties? +
Only what you opt into. Page DOM and goals go to your chosen LLM provider (Gemini, Claude, OpenAI). Issues go to your tracker (GitHub/Jira/Linear) only if you tick the box. Skopix itself has no telemetry, no analytics, no phone-home. Reports stay on your machine. Credentials never leave your machine.
How do I set Skopix up for a team? +
Run with the --team flag, a random SKOPIX_SECRET_KEY, and --host 0.0.0.0 so other machines can connect:

SKOPIX_SECRET_KEY="long-random-string" skopix dashboard --team --host 0.0.0.0 The first person to visit /setup becomes the admin. From there, go to the Users page in the dashboard and generate invite links for your teammates. They visit the link, set a password, they're in.
Will my single-user setup keep working after I add team mode? +
Yes — team mode is fully opt-in. Run skopix dashboard without the --team flag and it behaves exactly as it always did: no login, no users, no database. Run it with --team and you get the multi-user dashboard. They're completely independent code paths. Your existing tests, suites, and reports keep working in both modes.
How do user-level tokens work alongside workspace tokens? +
Workspace tokens in .skopix.env are the team-wide defaults — everyone uses them unless they've set their own. Each user can override any token from their "My settings" page in the dashboard (encrypted AES-256-GCM, stored in SQLite). When a test runs, Skopix checks the user first, falls back to workspace if not set. So Bob's GitHub issues are created as Bob (using his token) and Alice's as Alice. Same workspace, same suites, different attribution.
What happens if I lose or change my SKOPIX_SECRET_KEY? +
The secret key is what encrypts users' personal tokens. If you change it, existing stored tokens can't be decrypted — Skopix will fall back to workspace defaults and users will need to re-enter their own tokens. Treat the secret key like a database password. Store it somewhere safe (1Password, your infra's secret manager, an encrypted env file). Logins still work — only the stored tokens are affected.
Can a viewer run tests? +
No. Viewers are strictly read-only — they see the dashboard, reports, and issues but can't run tests, edit suites, or push to trackers. The UI hides write controls so they don't get misleading buttons that fail. If a viewer needs to verify something, ask an admin to upgrade them to Editor. Editors can do everything except manage users.
How do I revoke someone's access? +
As admin, go to the Users page in the dashboard. You can disable an account (all their sessions immediately invalidated, can't log in, but their history is preserved) or remove them entirely (account deleted, but their past test runs and reports remain visible to the team). The dashboard also prevents you from removing the last remaining admin so you can't lock yourself out.
Where is my data stored, exactly? +
Everything is on the machine running Skopix:

User accounts & sessions → ~/.skopix/skopix.db (SQLite file)
Encrypted user tokens → same SQLite file, encrypted with AES-256-GCM
Workspace credentials → ~/.skopix/credentials.yaml (chmod 0600)
Test reports + videos → ./skopix-reports/ in your working directory
Suite definitions → *.suite.yaml files in your working directory (commit to git)
Workspace tokens → .skopix.env in your working directory (do NOT commit)

To back up Skopix: archive the SQLite file + the reports directory + the suite YAMLs. To migrate to a new server: copy the same things.
How do remote teammates connect when I'm on Wi-Fi at home? +
Three realistic options. Same network (office Wi-Fi): they connect at http://your-ip:9000 directly. Remote teammates: use a mesh VPN like Tailscale (free for up to 100 devices, takes 5 minutes to set up). Always-on access: deploy to a small cloud VM (~£4/mo at Hetzner) where everyone can reach it from anywhere.
Can I change my password? +
Yes — every user has a "My settings" page in the dashboard with a Change Password form. Requires entering your current password to confirm it's you. Minimum 8 characters. Your existing session stays active after a password change. If you forgot your password entirely, an admin can disable your account and re-invite you. There's no public password-reset flow because that'd require SMTP — keeping things infrastructure-free.
How do I deploy with Docker? +
One command:

docker run -d --name skopix -p 9000:9000 -v ~/skopix-data:/data -e SKOPIX_SECRET_KEY="$(openssl rand -base64 32)" -e SKOPIX_TEAM_MODE=true ghcr.io/x444dyx/skopix:latest Multi-arch image (amd64 + arm64) on GitHub Container Registry. All persistent data lives in ~/skopix-data — back that up and you've backed up Skopix entirely. For production with auto-HTTPS, use the docker-compose + Caddy recipe in the Self-host section above, or see the full DOCKER.md guide.
How do I update to a new version? +
Docker (recommended):
docker compose pull && docker compose up -d Pulls the latest image and recreates the container. Your data in ./skopix-data is preserved.

npm install: npm install -g skopix@latest and restart the dashboard.

To pin to a specific version, change :latest to a version tag like :1.0.0 in your docker-compose.yml. Available tags are listed on the GitHub Container Registry page.
Does Skopix work behind nginx/Caddy/Traefik? +
Yes — Skopix is a vanilla HTTP service on port 9000. It works behind any reverse proxy. We ship a Caddy recipe in the Self-host section because Caddy's auto-HTTPS is the simplest path. If you already run nginx/Traefik/Cloudflare Tunnel for other services, just point them at skopix:9000 (Docker) or http://your-server:9000 (bare metal). HTTPS termination happens at the proxy, not in Skopix.
